SMA — Intranet maintenance automobile en cours

Repo : github.com/christiandrochon/sma-eventsourcing-17

Objectif

SMA est une application intranet de maintenance automobile conçue comme un projet “terrain”. L’objectif n’est pas seulement de coder des écrans : c’est de construire un socle clair, évolutif et traçable, capable d’accueillir des règles métier et des évolutions sans dérive de dette technique.

Ce qui existe aujourd’hui

  • Backend : Spring Boot (WebFlux)
  • Frontend : Thymeleaf (rendu côté serveur)
  • Base de données : PostgreSQL
  • Sécurité : OAuth2 / OpenID Connect via Keycloak (selon environnement)
  • Traçabilité : CQRS + Event Sourcing (Axon)
  • Exécution locale : Docker Compose (build des images à la volée via Dockerfile)

Exécution locale (démo reproductible)

Depuis la racine du projet :

docker compose -f compose.yaml up -d

Accès services (local)

  • Frontend (Thymeleaf) : http://localhost:8091
  • Backend : http://localhost:8092
  • Healthcheck : http://localhost:8092/actuator/health
  • Axon Server : http://localhost:8024
  • pgAdmin : http://localhost:6002
  • PostgreSQL : localhost:5432

Architecture

Le projet applique CQRS (séparation lecture/écriture) et Event Sourcing (journalisation métier), afin de rendre les changements auditables et de faciliter l’évolution du modèle. Le choix Thymeleaf (SSR) permet de garder une interface simple, proche du domaine, avec validation/binding Spring.

Profils Spring

  • default : configuration par défaut
  • local : exécution locale hors Docker
  • prod : exécution via Docker Compose

Roadmap (intention, pas “déjà fait”)

  • RBAC avancé (rôles/permissions alignés avec le métier)
  • RGPD by design (minimisation, traçabilité, droits, purge)
  • Recherche : PostgreSQL + index (et moteur si nécessaire)
  • Découpage services si le périmètre l’exige (sans microservices “pour faire joli”)
  • Prédiction (ML/Python) quand il y a un vrai use-case (maintenance préventive, anomalies, etc.)
