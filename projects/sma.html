SMA — Intranet maintenance automobile en cours

Repository GitHub

Objectif

SMA est une application intranet de maintenance automobile conçue comme un projet "terrain".
L’objectif est de construire un socle technique clair, traçable et évolutif capable d’accueillir des règles métier sans dérive de dette technique.

Stack actuelle

  • Backend : Spring Boot (WebFlux)
  • Build : Maven
  • Frontend : Thymeleaf (server-side rendering)
  • Base de données : PostgreSQL
  • Sécurité : OAuth2 / OpenID Connect via Keycloak
  • Traçabilité métier : CQRS + Event Sourcing (AxonIQ)
  • Exécution locale : Docker Compose (images construites via Dockerfile et Maven)

Architecture

Le projet applique les principes CQRS (séparation des modèles de lecture et d’écriture) et Event Sourcing (journalisation des événements métier). Cette approche permet de conserver un historique complet des changements, d’améliorer l’auditabilité et de faciliter l’évolution du modèle métier.

Le choix de Thymeleaf (rendu côté serveur) permet de conserver une interface simple, fortement intégrée avec l’écosystème Spring (validation, binding, sécurité), tout en évitant la complexité d’un frontend SPA lorsque cela n’est pas nécessaire.

Exécution locale

  • Démarrer : docker compose -f compose.yaml up -d
  • Arrêter : docker compose -f compose.yaml down

Services disponibles en local

  • Frontend (Thymeleaf) : http://localhost:8091
  • Backend API : http://localhost:8092
  • Healthcheck : http://localhost:8092/actuator/health
  • Axon Server : http://localhost:8024
  • pgAdmin : http://localhost:6002
  • PostgreSQL : localhost:5432

Profils Spring

  • default : configuration par défaut
  • local : exécution locale hors Docker
  • prod : exécution via Docker Compose

Roadmap

  • RBAC avancé (gestion des rôles et permissions alignée avec le modèle métier)
  • Conformité RGPD : minimisation des données, traçabilité, droits d’accès et purge
  • Recherche avancée : index PostgreSQL et intégration d’ElasticSearch si nécessaire
  • Découpage éventuel des services si l’évolution du périmètre le justifie (sans microservices prématurés)
  • Prédiction (ML / Python) en présence d’un cas d’usage réel : maintenance préventive, détection d’anomalies, etc.
  • Reconnaissance visuelle des symboles du tableau de bord via une application mobile pour assister au diagnostic
  • Application mobile dédiée au suivi et à l’analyse prédictive de l’état des véhicules
